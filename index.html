architecture issues
critical
Duplicated webhook handlers
WhatsAppWebhookView and WhatsAppPipelineWebhookView share ~80% identical logic (reset triggers, pagination, session expiry, greeting intercepts). A single bug fix must be applied twice, causing drift.
# Merge into one view; extract shared logic: class BaseWhatsAppHandler: def handle_reset(self, sender, conv): ... def handle_pagination(self, sender, conv): ... def handle_greeting(self, sender, conv): ...
critical
Dangling try block (dead code)
In WhatsAppPipelineWebhookView._handle_incoming around line ~800, there is a bare try: block with indented code but no matching conditional — the pagination handler was refactored but the old try/except orphan was left behind. This will silently run the wrong branch.
# Around "── Old handlers removed ──" comment: # The indented `try:` block calls _generate_ai_response # but has no triggering condition — remove or wrap: if False: # TODO: remove this dead block entirely try: ...
critical
god-function _generate_ai_response
This 300+ line function embeds a full QA flow, Gemini NLU, regex fallbacks, session management, and result routing. It cannot be tested in isolation, and any change risks breaking all downstream consumers (web chat, WA webhook, expand-search).
# Split into testable units: class QAOrchestrator: # mandatory Q flow class EntityExtractor: # Gemini + regex NLU class SessionQueryBuilder: # composed query # _generate_ai_response becomes a thin coordinator
warning
Dual pagination storage (session + DB)
Search results are written to both session_data["wa_pagination_state"] (Redis/cache) AND ConversationPipelineSession.search_results_json (Postgres). When they diverge the user gets inconsistent "load more" results. The session copy is the one used on the web channel; the DB copy is used on WhatsApp.
# Choose one source of truth: # Option A: always use DB (ConversationPipelineSession) # Option B: always use session cache # Remove the other write path entirely
warning
Phase 7 silently skipped on WhatsApp
Both WA webhooks detect QUERY_BUILDER phase and force-advance to RESULTS. This is correct for UX, but the check happens after orch.process() which may have already side-effected the DB. The skip should be a pre-process guard, not a post-hoc repair.
# In PipelineOrchestrator.process(): if self.is_whatsapp and phase == QUERY_BUILDER: self.session.current_phase = RESULTS return PhaseResult(handled=False) # Not: check after process() and overwrite
warning
Deep imports inside hot paths
from chat.models import ..., from chat.pipeline.orchestrator import ..., etc. appear inside _handle_incoming() which runs on every WA message. Python caches imports, but the deep nested imports increase cold-start time and obscure the dependency graph.
# Move all imports to module top-level: from chat.models import ( ChatConversation, ChatMessage, ConversationPipelineSession, PipelinePhase ) # Inside method: only local computation imports
warning
Phase 3 Gemini call is synchronous + blocking
_get_q3_2_suggestions() calls Gemini on every Phase 3 entry to generate price suggestions. If Gemini times out or returns an error, the user gets no budget prompt. The fallback exists but the Gemini call is not cached.
# Cache suggestions by (property_type, city, goal): @lru_cache(maxsize=256) def _get_q3_2_suggestions(prop, city, goal, mn, mx): ... # Or pre-compute for top-10 city/type combos at startup
info
_RESET_TRIGGERS and _GREETING_TRIGGERS duplicated
Both webhook views define identical frozenset constants inline. Any new greeting word must be added in two places.
# chat/constants.py WA_RESET_TRIGGERS = frozenset({"ابدأ", "reset", ...}) WA_GREETING_TRIGGERS = frozenset({"سلام", "hi", ...})
info
logger.warning used for INFO-level logs
Throughout both webhooks, diagnostic messages like [PAGINATION-DEBUG], [WA-CACHE], [WA-MESSAGE-START] use logger.warning() rather than logger.info() or logger.debug(). This pollutes warning-level monitoring alerts.
logger.debug("[PAGINATION-DEBUG] offset=%d", offset) logger.info("[WA-CACHE] cached %d results", n) # Reserve logger.warning() for actual unexpected states
info
No test coverage entry points visible
The phase handlers (Phase1Handler, Phase2Handler, etc.) are cleanly instantiable, but _generate_ai_response and _handle_incoming depend on DB, Redis, and Gemini with no injectable fakes. Adding factory fixtures for ConversationPipelineSession would unlock unit testing.
# conftest.py @pytest.fixture def pipeline_session(db): conv = ChatConversation.objects.create() return ConversationPipelineSession.objects.create( conversation=conv, current_phase=1 )
good
Phase handlers are clean and well-structured
Phases 1–4 and 7 each have a single-responsibility handle(session, user_text) entry point, clear state transitions, and good fallback behavior. The PhaseResult dataclass makes return types explicit.
good
Dedup guard prevents double-processing
mark_message_processed(message_id) is called before any logic runs, preventing WhatsApp retry storms from triggering duplicate AI calls and DB writes. Good resilience pattern.
good
Pagination persisted in DB for durability
Storing sorted search results in ConversationPipelineSession.search_results_json means pagination survives process restarts and Redis eviction. The sort-before-store pattern ensures consistent page ordering.
